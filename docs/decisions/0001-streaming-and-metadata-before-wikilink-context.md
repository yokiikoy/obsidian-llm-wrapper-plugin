# ADR 0001: ストリーミング＋リッチメタデータ（P1+P2）を Wikilink コンテキスト（P3）より先に実装する

## Status

Accepted

## Context

- LLM ラッパー相当機能のギャップ一覧があり、そのうちストリーミング・キャンセル・usage / finish_reason / reasoning の可視化（P1+P2）と、Vault 内 `[[wikilink]]` からの本文連結（P3）が候補となった。
- P3 は送信ペイロードを増やし、**非ストリーミングのブロッキング UI** と組み合わせると待ち時間が悪化しやすい。
- P1 と P2 はストリーム終端でメタデータを回収する**一体設計**が望ましく、プロトコル層の二重実装を避けたい。

## Decision

1. **最優先:** P1（ストリーミング、`ReadableStream` / SSE 等のプロバイダ準拠パース、`AbortController`）と P2（完了時の `usage`、`finish_reason`、API が提供する範囲での reasoning 分離）を**同一フェーズで実装**する。
2. **その後:** P3（Wikilink コンテキスト解決）を実装する。
3. **採用しない順序:** P3 を P1+P2 より先に実装する（ペイロード増大 × ブロッキングの相乗を避けるため）。

## Consequences

### 良い点

- コンテキスト量が増える前に、逐次表示とキャンセル可能なパイプラインを用意できる。
- メタデータ回収をストリーム完了イベントに集約し、後からの拡張で二重メンテしにくい。

### 悪い点・リスク

- P1+P2 の実装負荷が一塊になり、Gemini / DeepSeek でパース経路が二系統必要になる。
- reasoning の形はモデル・API 依存のため、実装時に実レスポンスで仕様を確定する必要がある。
